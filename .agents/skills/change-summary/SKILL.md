---
name: change-summary
description: 当任务是总结本次代码改动、整理评审说明、生成提交前摘要、撰写开发记录或输出结构化变更说明时使用。适用于已有实际代码修改，需要基于当前改动生成清晰总结的场景。不适用于从零设计方案，或在没有实际改动的情况下臆测变更内容。
---

目标：
根据本次实际改动，输出清晰、可复用、可直接用于 review、提交说明、开发记录或阶段性汇报的变更摘要。

仓库背景：

- 本仓库是 Vite + React + TypeScript 项目。
- 默认遵循最小改动原则。
- 变更总结必须基于真实改动、真实文件和真实验证结果，不得把猜测写成已完成事实。
- 如果存在环境限制、沙箱限制、提权重跑或未完成验证，必须如实写明。
- 总结应服务于沟通，而不是堆砌细节；既要准确，也要让人快速看懂。

适用场景：

- 本次已经完成一轮代码修改，需要整理改动摘要
- 需要输出可直接用于 review 的变更说明
- 需要整理提交前说明
- 需要记录阶段性开发进展
- 需要把修改文件、验证结果、风险点统一汇总
- 需要把本次改动说明同步给他人

不适用场景：

- 还没有实际代码改动，只是在讨论方案
- 需要从零设计一套新功能方案
- 需要生成完整 spec 或正式设计文档
- 需要编写提交命令或直接执行 git 操作
- 需要对没有检查过的代码内容做推测性总结

执行原则：

1. 先读取本次实际修改过的文件和相关上下文。
2. 只基于真实改动进行总结，不臆测未实现内容。
3. 每个关键文件都要说明：
   - 改了什么
   - 为什么改
   - 对哪个界面、逻辑或问题有影响
4. 如果本次改动是为了解决具体问题，要明确写出：
   - 原问题是什么
   - 采用了什么修复方式
   - 当前结果如何
5. 如果已有验证结果，必须纳入总结。
6. 如果验证过程中出现环境限制、PowerShell 执行策略、沙箱限制、提权重跑或 `spawn EPERM` 一类问题，要区分：
   - 默认环境结果
   - 排除环境限制后的最终结果
7. 如果存在未完成部分、待确认部分或已知局限，要单独列出，不要藏在正文里。
8. 输出应优先服务于以下用途：
   - review 说明
   - 提交前摘要
   - 开发记录
   - 阶段性同步
9. 不要把很小的样式微调写成“大范围优化”，也不要把局部修复夸大成系统性重构。
10. 用语应准确、克制、可复核。

推荐工作流：

1. 先确认本次改动的目标。
2. 列出本次涉及的关键文件。
3. 按文件归纳每个文件的修改内容和修改目的。
4. 总结本次改动整体解决了什么问题。
5. 整理本次验证结果。
6. 提炼当前剩余风险、限制和后续建议。
7. 按固定结构输出最终摘要。

总结要求：

- 优先按“整体概览 + 文件级说明 + 验证结果 + 风险与后续”组织。
- 文件级说明不必机械罗列所有小改动，但关键改动必须覆盖。
- 如果多个文件共同完成同一个目标，可以先给整体结论，再说明各文件分工。
- 若本次改动与之前的 skill 链路有关，可以自然提及：
  - 修复了什么问题
  - 文档是否已同步
  - 验证结论如何
- 若无验证结果，不得写成“已验证通过”。
- 若验证只在提权后通过，必须明确写出这一点。

输出格式要求：
请在完成后按下面结构汇报：

1. 本次改动概览

- 本次改动主要解决了什么问题
- 改动范围大致涉及哪些模块

2. 关键文件修改说明

- 按文件列出关键改动
- 每个文件说明：
  - 改了什么
  - 为什么改
  - 影响了哪里

3. 验证结果

- 列出本次已有验证
- 明确区分：
  - 默认环境结果
  - 如有提权、切换环境或重跑后的结果
- 如果没有完成验证，要如实说明

4. 当前结论

- 当前是否达到“可继续 review / 可继续开发 / 可考虑提交”的状态
- 当前结论基于哪些事实
- 是否还有明显待确认事项

5. 已知限制与风险

- 当前还存在哪些限制
- 是否有环境级限制、沙箱限制或未覆盖验证
- 是否存在后续可能继续优化的点

6. 建议的下一步

- 下一步最合理的动作是什么
- 是继续修复、继续验证、同步 spec，还是准备提交

完成标准：

- 总结基于真实改动
- 结构清晰
- 不混入臆测内容
- 能直接复用到 review、提交说明或开发记录
- 能区分代码事实、验证事实和环境限制
