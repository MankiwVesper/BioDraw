---
name: repo-explore
description: 当任务是梳理仓库结构、查找某个功能相关文件、理解模块职责、分析调用链、定位入口文件、追踪数据流或为后续修改做代码探索准备时使用。适用于先理解仓库、暂不改代码的场景。不适用于直接修复 bug、直接改代码、直接同步 spec 或直接做提交准备。
---

目标：
在不修改代码的前提下，帮助用户快速弄清楚当前仓库中某个功能、模块或问题相关的代码分布、职责划分、调用关系和后续切入点。

仓库背景：

- 本仓库是 Vite + React + TypeScript 项目。
- 代码探索的目标是“帮助理解和定位”，不是直接实施修改。
- 输出应尽量服务于后续开发、调试、重构、spec 同步或验证，而不是只机械罗列文件名。
- 探索过程默认只读，不主动修改文件，不主动执行会改变仓库状态的命令。
- 如果信息不足，应明确说明“当前只能初步判断”，不要把推测写成确定事实。

适用场景：

- 想找某个功能相关代码分布在哪些文件
- 想确认某个界面、面板、控件的入口文件在哪里
- 想梳理某个模块的职责划分
- 想分析对象选中、状态更新、属性传递、事件响应的数据流
- 想为后续改代码、写 spec、做验证先做代码探索
- 想快速知道“应该先看哪些文件”

不适用场景：

- 已经明确知道要改哪里，准备直接改代码
- 已经有明确 TypeScript 报错，需要直接修复
- 需要直接更新 spec
- 需要直接输出提交说明
- 需要直接跑验证命令或执行 git 操作
- 需要做大型架构设计，而不是基于现有代码探索

执行原则：

1. 先明确本次探索目标。
2. 只围绕目标阅读直接相关的代码，不要无边界扩散。
3. 优先定位：
   - 入口文件
   - 核心组件
   - 状态来源
   - 关键调用链
   - 布局或样式控制点
4. 如果一个功能涉及多个文件，应说明每个文件的大致职责，而不是只给文件名。
5. 如果存在明显的数据流或调用链，应按顺序解释：
   - 从哪里进入
   - 中间经过哪些关键层
   - 最终影响到哪里
6. 如果某个判断带推测成分，要明确标注“初步判断”或“推测”。
7. 不在探索阶段顺手提出大规模改造方案，除非用户明确要求。
8. 输出应优先帮助用户回答：
   - 相关代码在哪
   - 各文件分别负责什么
   - 后续应该先改哪
   - 如果要继续深入，下一步该看什么

推荐工作流：

1. 先复述本次要探索的目标。
2. 搜索并定位与目标最直接相关的文件。
3. 阅读入口文件和关键中间层文件。
4. 梳理组件关系、状态传递或调用链。
5. 提炼每个关键文件的职责。
6. 总结当前理解结果。
7. 给出后续建议的切入点，但不直接修改代码。

探索重点参考：

- 如果目标是界面或面板类功能，优先找：
  - 页面入口
  - 布局容器
  - 面板组件
  - 样式和布局控制位置
- 如果目标是交互类功能，优先找：
  - 事件触发点
  - 状态更新逻辑
  - props 传递链
  - store / context / 上层页面状态来源
- 如果目标是渲染类功能，优先找：
  - render 入口
  - 渲染分发逻辑
  - 具体 renderer 文件
- 如果目标是对象编辑类功能，优先找：
  - 选中态来源
  - 当前对象数据来源
  - Inspector 渲染入口
  - 更新回写路径

输出格式要求：
请按下面结构输出：

1. 本次探索目标

- 本次要梳理的功能、模块或问题是什么
- 当前探索范围是什么

2. 相关关键文件

- 列出本次最关键的文件
- 每个文件说明：
  - 文件路径
  - 大致职责
  - 为什么它和本次目标相关

3. 调用链 / 数据流 / 结构关系

- 说明关键调用链、组件关系或数据传递路径
- 如果无法完全确认，要明确说明哪一段仍需继续确认

4. 当前结论

- 当前已经可以确定什么
- 哪些点只是初步判断
- 如果后续要修改代码，优先应该从哪里下手

5. 建议的下一步

- 建议继续阅读哪些文件
- 建议接下来使用哪个 skill 更合适：
  - ts-error-fixer
  - spec-sync
  - ui-change-verify
  - change-summary
  - git-commit-prep
- 如果暂时还不建议动代码，要明确说明原因

完成标准：

- 能帮助用户快速理解相关代码分布
- 能指出入口文件、关键文件和职责划分
- 能梳理基本调用链或数据流
- 不混入未验证的确定性结论
- 输出能直接作为后续开发或继续提问的基础
