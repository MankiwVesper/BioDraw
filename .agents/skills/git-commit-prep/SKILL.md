---
name: git-commit-prep
description: 当任务是为本次实际代码改动准备提交说明、整理 commit message 候选、生成提交前摘要或确认提交范围时使用。适用于已经完成代码修改、验证和总结，需要把结果整理成可提交说明的场景。不适用于直接执行 git 提交，也不适用于在没有真实改动的情况下臆造提交内容。
---

目标：
根据本次真实改动，生成清晰、准确、可直接用于 git commit 或提交前确认的说明内容。

仓库背景：

- 本仓库是 Vite + React + TypeScript 项目。
- 默认遵循最小改动原则。
- 提交说明必须基于真实改动、真实文件和真实验证结果。
- 不得把未完成的工作写成已完成事实。
- 不直接执行 git commit，不直接代替用户提交代码。
- 输出应便于用户快速判断：现在是否适合提交，如果提交，应如何描述本次改动。

适用场景：

- 本次代码已经改完，需要整理提交说明
- 需要生成 commit message 候选
- 需要在提交前确认改动范围是否清晰
- 需要把验证结果与已知限制一起纳入提交说明
- 需要把本次修改总结为简洁、准确、可复用的提交文本

不适用场景：

- 还没有完成实际代码改动
- 尚未确认本次改动解决了什么问题
- 需要直接执行 git add / git commit / git push
- 需要生成完整 spec、设计文档或 PR 长文说明
- 没有真实验证结果却想伪装成“已验证完成”

执行原则：

1. 先读取本次真实改动及其上下文。
2. 优先基于实际修改文件、验证结果和变更目标整理内容。
3. commit 说明要准确、克制、可复核，不夸大改动范围。
4. 若本次改动只是局部修复，不得写成“大规模重构”。
5. 若存在环境限制、沙箱限制、提权后通过等情况，要在提交前说明里如实体现，但不要把环境问题误写成代码缺陷。
6. 若验证尚未完成，应明确提示“暂不建议提交”或“建议补充验证后再提交”。
7. 输出内容要区分：
   - 简短 commit message
   - 稍详细的提交说明
   - 当前是否建议提交

推荐工作流：

1. 先总结本次改动目标。
2. 列出本次关键改动文件。
3. 归纳本次改动类型：
   - 修复
   - 优化
   - 文档同步
   - 验证调整
4. 结合验证结果，判断当前是否达到“可提交”状态。
5. 生成多种 commit message 候选。
6. 给出一版更完整的提交说明。
7. 明确提示当前是否建议提交。

输出格式要求：
请按下面结构输出：

1. 提交建议结论

- 当前是否建议提交
- 做出该判断的依据是什么

2. 推荐 commit message

- 给出 3 条候选
- 风格要求：
  - 简洁
  - 准确
  - 不夸大改动范围

3. 推荐详细提交说明

- 本次改动主要解决了什么问题
- 涉及哪些关键文件或模块
- 当前验证结果如何
- 是否存在已知限制或待继续确认项

4. 提交前提醒

- 是否还有建议补做的验证
- 是否还有不适合一并提交的改动
- 是否建议先同步 spec、整理文件或拆分提交

完成标准：

- 内容基于真实改动
- 能直接用于提交前确认
- commit message 简洁可用
- 提交说明准确不夸大
- 能明确指出当前是否适合提交
