---
name: ui-change-verify
description: 当任务涉及 React、TypeScript、Inspector、编辑器面板、对象列表、布局样式、属性面板或局部交互调整后，执行最基本且可靠的项目验证，并按固定结构输出结果。适用于前端局部修改后的快速验收。不适用于后端、部署、文档写作或大型端到端测试。
---

目标：
对本次前端改动进行最小但可靠的验证，输出可直接用于 review、提交前检查或后续沟通的结果说明。

仓库背景：

- 本仓库是 Vite + React + TypeScript 项目。
- 默认遵循最小改动原则。
- 验证应围绕“本次改动实际影响范围”展开，不要无边界扩散。
- 如果无法完成某项验证，必须明确说明原因，不得假装已验证。
- 当前开发环境可能涉及 Windows、VS Code 中的 Codex 插件、PowerShell、沙箱限制以及提权重跑等差异，验证结论必须区分“代码问题”和“执行环境问题”。

适用场景：

- 修改了 React 组件
- 修改了 TypeScript 类型或 props
- 调整了 Inspector 面板布局
- 调整了对象列表、搜索栏、筛选栏、输入框、间距、滚动区
- 修改了编辑器页面中的局部 UI 或交互
- 需要在改动完成后做一次标准化验收

不适用场景：

- 后端接口开发
- 部署、CI/CD、服务器配置
- 纯文档更新
- 大型端到端自动化测试
- 性能压测
- 与当前前端改动无关的全局质量治理

验证原则：

1. 先识别本次改动涉及的模块和影响范围。
2. 优先做“成本低、价值高”的基础验证。
3. 不为了显得全面而运行大量无关检查。
4. 如果某个命令失败，要提炼核心原因，不要只堆原始日志。
5. 如果某项验证无法执行，要说明：
   - 为什么无法执行
   - 这会带来什么验证缺口
6. 不得宣称“已验证正常”除非确实完成了对应验证。
7. 如果失败原因明显属于权限、沙箱、执行策略、PowerShell、临时目录或配置加载阶段的环境限制，应优先判定为“验证被环境阻塞”，而不是直接认定为代码回归。
8. 如果同一命令在提权、批准后重跑、切换执行环境或改用等价调用方式后能够通过，应将代码验证视为通过，同时单独记录默认环境存在限制。
9. 对于环境问题与代码问题并存可能性较高的场景，要明确区分“默认环境结果”和“排除环境干扰后的最终结果”。

推荐工作流：

1. 先总结本次改动影响范围。
2. 在 Windows 环境下优先运行：
   - npm.cmd run build
3. 如果仓库可用且成本合适，再运行：
   - npm.cmd run lint
4. 如果默认环境下验证失败，先判断失败是否属于环境或沙箱限制。
5. 如有必要，在获得批准或切换环境后重跑关键验证命令。
6. 补充说明建议人工检查的界面区域。
7. 最后按固定格式输出验证结果。

命令使用要求：

- 优先使用仓库现有命令。
- 在 Windows 或 PowerShell 环境下，优先使用 `npm.cmd`，不要默认使用 `npm`，以避免 `npm.ps1` 执行策略带来的干扰。
- 如果命令不存在、报错或未配置，要如实说明。
- 不自行发明仓库中没有的验证命令。
- 不把“代码看起来没问题”当作“验证通过”。
- 如果 `vite build` 在读取配置或加载 `vite.config.ts` 阶段出现 `spawn EPERM`、权限拒绝、沙箱阻塞等环境级错误，应优先判定为“验证被环境或沙箱限制阻塞”。
- 如果同一构建命令在提权、放宽权限、改用 `npm.cmd`、批准后重跑或切换执行环境后能够通过，应将本次代码验证视为通过，同时单独记录默认环境存在权限或沙箱限制。
- 如果 lint 通过而 build 仅在默认受限环境下失败，不应直接下结论说“本次改动构建失败”；应说明“默认环境未完成验证，但排除环境因素后构建通过”。
- 若某条命令只能在提权后通过，必须在结果中明确写出“默认环境结果”和“提权后结果”，不得混写。

人工检查建议：
当本次改动属于下列情况时，除了命令验证，还应建议人工查看相关区域：

- 面板对齐、间距、滚动行为变化
- 输入框高度、行内排版、字段并排展示变化
- Inspector 区块折叠/展开或 section 布局变化
- 左侧对象列表、搜索框、筛选器布局变化
- 颜色选择器、显隐/锁定等交互控件变化

输出格式要求：
请在完成后按下面结构汇报：

1. 本次验证范围

- 本次主要验证了哪些模块或文件相关改动
- 哪些方面属于本次验证覆盖范围

2. 已执行验证

- 列出实际运行的命令
- 每条命令分别说明：
  - 默认环境结果
  - 如有提权/切换环境后的结果
- 如失败，提炼失败原因

3. 建议人工检查项

- 哪些界面或交互建议人工再确认
- 为什么需要人工确认

4. 当前结论

- 当前是否达到“可继续 review / 可继续开发 / 可考虑提交”的状态
- 当前结论是基于默认环境、提权后环境，还是两者综合判断
- 是否仍存在明显风险

5. 剩余风险与后续建议

- 还有哪些未覆盖验证
- 是否存在环境级限制尚未解决
- 下一步最合理的检查或修复动作是什么

完成标准：

- 验证范围清楚
- 实际执行情况如实说明
- 能区分代码问题与环境问题
- 不夸大验证结论
- 输出内容可直接用于 review 或提交前确认
