---
name: ts-error-fixer
description: 当任务是修复 TypeScript 报错、React + TypeScript 编译错误、类型不匹配、导入导出错误、属性缺失、泛型或联合类型报错时使用。适用于已有明确报错信息，目标是定位相关文件、做最小范围修改并验证结果。不适用于新增大功能、跨模块重构、纯样式优化或没有明确报错的开放式重构。
---

目标：
修复当前仓库中的 TypeScript 相关报错，并在结束前完成最基本的验证。

仓库背景：

- 本仓库是 Vite + React + TypeScript 项目。
- 默认遵循最小改动原则。
- 不要为了“顺手优化”而扩散改动范围。
- 不要修改与当前报错无关的 UI、样式、文案或架构。

适用场景：

- VS Code Problems 面板中的 TypeScript 报错
- `npm run build` 中的 TypeScript 编译报错
- 组件 props 类型不匹配
- import / export 错误
- 类型定义缺失
- `boolean | undefined` 不能赋给 `boolean` 之类的收窄问题
- React 组件参数、事件类型、对象字段类型报错

不适用场景：

- 新增一整套新功能
- 全局重构目录结构
- 纯 CSS / 布局美化
- 没有明确错误信息，只是想“代码更优雅”
- 后端、部署、CI、文档写作类任务

执行原则：

1. 先读取用户给出的报错信息。
2. 从报错中提取：
   - 文件路径
   - 行号
   - 错误码
   - 报错核心原因
3. 只打开和当前报错直接相关的文件。
4. 如果多个文件都被同一个错误链影响，优先从报错源头文件开始处理。
5. 优先做最小修改，不做无关重命名、不做风格统一、不做顺手重构。
6. 修改时保持现有命名风格、目录结构和代码组织方式。
7. 若存在多种修法，优先选择：
   - 改动范围更小
   - 对现有功能影响更低
   - 可读性足够但不过度设计
8. 如果发现报错本质上来自错误的调用方式，优先修调用方，而不是粗暴放宽类型。
9. 非必要不要使用 `any`。
10. 非必要不要删除已有类型约束。
11. 如果必须做权衡，明确说明为什么这样改。

推荐工作流：

1. 复述当前准备修复的报错。
2. 定位相关文件。
3. 给出简短修复思路。
4. 实施修改。
5. 运行适合当前仓库的验证命令。
6. 汇报修改文件、修改原因、验证结果、剩余风险。

验证要求：

- 优先运行：
  - npm run build
- 如果仓库可用且代价不高，再考虑：
  - npm run lint
- 如果某条命令无法运行或当前仓库未配置，明确说明，不要假装已验证。

输出格式要求：
请在完成后按下面结构汇报：

1. 修复对象

- 本次处理了哪些报错
- 根因是什么

2. 修改文件

- 列出修改过的文件
- 每个文件做了什么调整

3. 验证结果

- 运行了哪些命令
- 命令是否通过
- 是否还有剩余报错

4. 风险与后续

- 是否有潜在连带影响
- 是否建议继续做后续清理

完成标准：

- 当前目标报错被修复或明显收敛
- 没有引入新的无关改动
- 验证结果被如实说明
- 输出内容可用于直接 review
